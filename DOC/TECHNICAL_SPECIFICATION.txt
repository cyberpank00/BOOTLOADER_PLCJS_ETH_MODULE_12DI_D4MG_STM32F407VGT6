Цель - bootloader
Разработать автономный bootloader для устройства на STM32F407VGT6, который:
- запускается раньше основного приложения
- принимает решение: переходить в приложение или оставаться в режиме bootloader
- обеспечивает безопасное обновление firmware по Modbus TCP
- защищён от потери питания и обрыва связи во время обновления
- отображает минимальный статус через LED 
- допускает дальнейшее сопровождение в промышленной эксплуатации

1. Назначение и область применения
Bootloader предназначен для:
- первичного запуска устройства при отсутствии валидного приложения
- сервисного и удалённого обновления firmware по сети Ethernet
- восстановления устройства после неудачного обновления
- безопасной записи нового образа во внутреннюю Flash-память микроконтроллера

Устройство применяется как промышленный модуль обмена данными по Modbus TCP, поэтому bootloader должен быть:
- устойчивым к сетевым ошибкам
- предсказуемым по времени старта
- простым в диагностике
- безопасным по отношению к рабочему приложению

2. Основные требования к bootloader
2.1 Условия запуска bootloader
Bootloader должен оставаться активным при выполнении хотя бы одного из условий:
1. Нажата сервисная кнопка при старте устройства
2. Основное приложение отсутствует
3. Основное приложение невалидно
4. Во Flash установлен флаг запроса входа в bootloader
5. Процесс установки новой прошивки был прерван ранее и требует завершения или восстановления

Если ни одно условие не выполняется, bootloader должен передавать управление основному приложению

2.2 Индикация
Bootloader должен поддерживать:
- мигание одного светодиода с разным режимом для состояний:
- ожидание
- приём прошивки
- установка прошивки
- ошибка

2.3 Сетевой интерфейс
Bootloader должен:
- инициализировать Ethernet-интерфейс
- запускать сервер Modbus TCP- поддерживать прикладной протокол обновления firmware поверх Modbus TCP
- работать без FreeRTOS, на основе конечного автомата и неблокирующего цикла

2.4 Обновление firmware
Bootloader должен:
- принимать новый образ прошивки по сети блоками
- сохранять образ во временную область хранения
- проверять целостность образа
- устанавливать образ в область основного приложения только после полной проверки
- корректно восстанавливаться после сбоя питания, reset или обрыва связи

2.5 Безопасность обновления
Bootloader не должен считать приложение валидным, пока:
- образ не принят полностью
- не подтверждена его целостность
- установка в application area не завершена
- не подтверждена корректность установленного образа

3. Ограничения и принципы проектирования
3.1 Отказ от OpenBLT
Для данного проекта OpenBLT не выбран как основа, потому что:
- обновление выполняется не по типовым транспортам OpenBLT, а по Modbus TCP- требуется собственный прикладной протокол передачи образа
- уже имеется связанный с продуктом сетевой стек/транспорт
- ключевая задача — алгоритм безопасного обновления, а не выбор готового универсального bootloader framework

3.2 Отказ от FreeRTOS в bootloader
Bootloader рекомендуется выполнять без FreeRTOS, потому что:
- требуется минимизировать сложность
- bootloader должен быть максимально предсказуемым
- задачи bootloader естественно решаются через один цикл и state machine
- RTOS усложняет переход в приложение, расход RAM и сопровождение

Допускается использование FreeRTOS только если существующий Ethernet/Modbus TCP код принципиально неотделим от RTOS-архитектуры, однако это считается нежелательным вариантом

4. Архитектура решения
Рекомендуемая архитектура состоит из следующих модулей:
- startup / boot entry — ранний старт, проверка причины входа в bootloader
- platform — тактирование, GPIO, watchdog, базовые таймеры
- app_control — проверка валидности приложения, переход в приложение
- flash_driver — стирание/запись/чтение внутренней Flash
- metadata_storage — хранение состояния обновления
- ethernet_stack — минимально необходимая сетевая инициализация
- modbus_server — Modbus TCP обработчик
- fw_update_protocol — команды обновления firmware поверх Modbus
- installer — перенос образа из staging в application
- indicator — LED

5. Рекомендуемая карта памяти
Для STM32F407VGT6 внутренний Flash обычно составляет 1 МБ. Практически рекомендуется заложить такую логическую структуру:
- Bootloader region — защищённая область в начале Flash
- Metadata region — отдельный сектор под служебные структуры
- Application region — область основного приложения
- Staging region — временное хранение нового firmware-образа

Ориентировочный пример логического деления:
- Bootloader: 64–128 КБ- Metadata: 1 сектор Flash
- Application: основная рабочая область
- Staging: область под вторую копию образа или максимально возможную его часть

Критически важная оговорка
Нужно проверить фактический размер текущего приложения из репозитория. Если приложение занимает слишком много памяти и во внутреннюю Flash не помещаются одновременно:
- текущее приложение
- staging-копия
- bootloader

тогда возможны три варианта:
1. уменьшить размер приложения
2. использовать внешний носитель под staging
3. перейти на обновление без полного staging, что хуже по надёжности

Для промышленного устройства предпочтителен вариант с полной staging-областью, если это позволяет память

6. Рекомендуемая модель обновления
Лучший вариант для этого проекта — двухэтапная схема:
1. Приём нового образа в staging region2. Проверка целостности staging-образа
3. Переключение состояния в режим установки
4. Копирование staging -> application
5. Проверка установленного application
6. Пометка application как валидного
7. Перезагрузка и запуск

Почему это лучший вариант
Такая схема защищает от ситуаций:
- пропало питание во время обновления
- TCP-сессия оборвалась
- был принят не весь файл
- был повреждён один из блоков
- произошёл reset во время установки

7. Состояния bootloader
Рекомендуется реализовать state machine со следующими состояниями:
- BOOT_START — начальная инициализация
- BOOT_CHECK_ENTRY — анализ причин входа в bootloader
- BOOT_WAIT_COMMAND — ожидание команды обновления по Modbus TCP
- BOOT_PREPARE_UPDATE — очистка staging и подготовка metadata
- BOOT_RECEIVE_FW — приём блоков прошивки
- BOOT_VERIFY_STAGING — проверка целостности всего образа
- BOOT_INSTALL_FW — копирование в application area
- BOOT_VERIFY_APP — проверка установленной прошивки
- BOOT_READY_TO_BOOT — готовность к запуску приложения
- BOOT_ERROR — состояние ошибки

8. Metadata и журнал состояния
В отдельной служебной области Flash должна храниться структура состояния, например:
- magic- structure_version- boot_state- app_valid- install_requested- image_size- image_crc32- block_size- block_count- received_block_count- received_bitmap_crc или bitmap блоков
- product_id- hw_revision- fw_version- install_in_progress- last_error_code- attempt_counter

Назначение metadata
Metadata нужна, чтобы после любого reset bootloader понимал:
- идёт ли приём образа
- завершён ли приём
- началась ли установка
- было ли прерывание питания во время установки
- можно ли запускать приложение

9. Проверка валидности приложения
Bootloader должен проверять приложение по нескольким критериям:
1. Первый вектор MSP указывает в допустимую область RAM
2. Reset Handler указывает в допустимую область Flash application
3. Заголовок приложения содержит корректный product_id4. Заголовок приложения содержит корректную длину и CRC
5. CRC приложения совпадает с ожидаемой

Простая проверка MSP и Reset Handler недостаточна для промышленного применения. Нужна также контрольная сумма всего образа

10. Формат образа firmware
Рекомендуется использовать собственный заголовок образа, размещённый в начале application image либо отдельно в metadata при загрузке. Заголовок должен содержать:
- magic- product_id- hw_revision- fw_version- image_size- image_crc32- vector_table_offset- reserved

Это позволит:
- не принимать прошивку от другого устройства
- сравнивать версии
- проверять совместимость оборудования
- быстро валидировать образ перед установкой

Рекомендуется не передавать прошивку как набор произвольных регистров без структуры, а определить чёткий прикладной протокол

12. Алгоритм безопасного обновления
Этап 1. Вход в bootloader
Инициализация минимальной платформы
Проверка кнопки
Проверка metadata
Проверка валидности приложения
Выбор: запуск приложения или переход в сервисный режим
Этап 2. Подготовка обновления
Получение BEGIN_UPDATE
Проверка параметров образа
Проверка, что размер образа помещается в staging
Очистка staging-области
Сброс bitmap принятых блоков
Сохранение metadata
Этап 3. Приём образа
Приём блоков по WRITE_BLOCK
Проверка индекса и смещения
Проверка CRC блока, если используется
Запись данных блока во Flash staging
Обновление bitmap/счётчиков
Ответ клиенту
Этап 4. Верификация staging
Проверка, что все блоки получены
Вычисление CRC32 всего staging-образа
Сравнение с ожидаемым CRC32
При успехе запись install_requested = 1
Этап 5. Установка
Запись boot_state = INSTALL
Стирание application area
Копирование staging -> application по страницам/секторам
Проверка CRC application
Проверка базовой валидности векторов
Запись app_valid = 1
Сброс install_requested
Сброс ошибок
Перезагрузка
Этап 6. Поведение после сбоя питания
Если питание пропало:
- во время RECEIVE_FW — старая прошивка ещё не затронута, можно продолжать update
- во время INSTALL_FW — приложение считать невалидным, установку повторить
- после успешного COMMIT — запускать новое приложение

13. Стратегия по CRC и контролю целостности
Рекомендуется использовать:
- CRC32 для всего образа — обязательно
- CRC блока — желательно
- контроль product_id / hw_revision — обязательно

Минимальный набор проверок:
- размер
- принадлежность образа данному устройству
- CRC staging
- CRC application после установки

14. Логика запуска приложения
Перед переходом в приложение bootloader обязан:
1. Остановить сетевую активность
2. Деинициализировать Ethernet/MAC/DMA
3. Отключить прерывания периферии
4. Остановить SysTick
5. Очистить pending interrupts при необходимости
6. Переключить VTOR на таблицу векторов приложения
7. Загрузить MSP из первого слова приложения
8. Передать управление на Reset Handler приложения

15. Индикация состояний
Рекомендуется следующая семантика LED:
- редкое мигание — ожидание команды
- частое мигание — идёт приём прошивки
- пачки миганий — установка
- постоянное свечение или особый шаблон — ошибка

16. Надёжность и эксплуатационные требования
Bootloader должен обеспечивать:
- восстановление после обрыва связи
- восстановление после перезапуска питания
- невозможность запуска частично установленной прошивки
- отказ от установки несовместимого образа
- диагностируемые коды ошибок

Рекомендуемые коды ошибок:
- несовместимый product_id
- несовместимая hardware revision
- неверный размер образа
- ошибка CRC блока
- ошибка CRC образа
- ошибка стирания Flash
- ошибка записи Flash
- ошибка валидации application
- тайм-аут update-сессии

17. Рекомендации по реализации без RTOS
Архитектура основного цикла bootloader:
1. ethernet_poll()2. modbus_poll()3. update_protocol_poll()4. installer_poll()5. indicator_poll()6. watchdog_poll()

Все операции должны быть неблокирующими, кроме допустимых коротких окон записи/стирания Flash

19. Что важно проверить по основному приложению из проекта
Перед окончательной реализацией bootloader нужно подтвердить по существующему приложению:
- фактический размер firmware
- используемый linker script
- стартовый адрес текущего приложения
- таблицу векторов и перенос VTOR- размер и размещение сетевых буферов
- использование внутренних Flash-секторов
- требования к MAC/IP настройкам в bootloader
- совместимость Ethernet-драйвера в среде bootloader

Это нужно, чтобы корректно выбрать:
- границы Bootloader/Application/Staging
- объём metadata
- схему инициализации сети
- допустимый размер блока передачи

21. Критические риски проекта
Основные риски:
- во внутренней Flash может не хватить места на полноценный staging
- Ethernet/Modbus стек bootloader может оказаться слишком тяжёлым
- текущий код приложения может быть слишком тесно связан с RTOS или общей архитектурой
- ошибка в layout Flash приведёт к повреждению приложения или metadata
- некорректная обработка reset во время установки приведёт к циклическому неуспешному старту
